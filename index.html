Pain points

Photo CC Wall Papers Wide
[cathy]

Drupal users that want to start contributing

[cathy]
  • documentation of starting to contribute is confusing and not concise
  • they find an issue,
    maybe with a patch, but they don't know what to do to help get the issue fixed
    (next steps not clear)
    issue summary too vague,
    no background information or clear explanation of problem
  • do not know who to ask questions
  • no cross link documentation and discussions regarding a topic of interest
    • right now you have to know the secret tags,
      search for tags,
      a g.d.o group,
      maybe an initiative handbook page to get more info about a topic
      (example: accessibility)

Open source contributors new to Drupal

[cathy]
  • they have to learn something old, new to them
    they dont like it
    they would like to use git-style pull requests or github
  • they can't edit in place
    the open work process is scary (we dont need to solve it, it is an asset)
    they would like to use github-style code review tools

Mentors

[cathy]
  • keep a list of issues a mentor wants to go back to
    • use browser bookmarks
    • gmail tags
    • no d.o favorites or lists
  • some of the same things as new contributors since mentors invest a lot of time exposing procedures to them, which could be automated

All contributors

[cathy]
  • do not see recognition of contribution on their d.o profile
  • reviewers not credited
  • search for issues is hard

Working on issues

Photo CC-BY Boris Baldinger
Before we can work on an issue, we have to find one that is suitable and applies to our interests and expertise. Also, an issue that other people care about to move it forward.

1290740

How to label, aggregate and expose issues, docs, forum posts and groups to Topic pages

Important because...
  • helps find issues to work on by topic
  • might indicate what is a priority right now
  • a mix of curated and automated content
  • recent activity from issues and groups
  • list of people and their expertise
  • curated summary and list of links
  • topic metadata
  • "like Meetup, but for issues"

I found an issue... what next?

  1. show all relevant tasks to an issue (customizable per project)
  2. indicate what tasks still need to be done
  3. expose instructions for tasks that need to be done
Must know about dreditor. Must know about tasks template. Must determine which of the HTML to uncomment. When done, delete? update?

2013222

Add "Issue tasks" to project issues
and correlate tasks
with handbook documentation

Instead of a manual process, it should be easy to expose tasks and automatically link to handbook pages. Helps people who land directly on an issue from a google search.
